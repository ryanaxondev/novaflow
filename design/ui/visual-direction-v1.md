# Purpose

This document defines the reusable visual language for NovaFlow.

Its purpose is to establish a consistent design direction before creating any high-fidelity interfaces.

It defines how approved product decisions are communicated visually while preserving the established information hierarchy and conversion flow.

Visual design supports product communication.

It does not redefine the product.

---

# Design Constraint

Visual design may improve communication,

but it must never change approved product decisions.

Specifically, it must not:

- introduce new sections,
- remove approved sections,
- change the information hierarchy,
- alter the conversion flow,
- redefine product messaging.

---

# Design Goals

NovaFlow should feel:

- Professional
- Calm
- Modern
- Reliable
- Clear
- Focused

The interface should communicate confidence through simplicity.

Visitors should trust the product before interacting with it.

---

# Visual Principles

The interface should emphasize:

- clarity before decoration,
- hierarchy before aesthetics,
- consistency before originality,
- readability before density,
- communication before visual effects,
- function before embellishment.

Every visual decision should make the product easier to understand.

---

# Layout

## Container

Use a centered content container.

Prefer generous horizontal whitespace.

---

## Grid

Desktop-first during this phase.

The layout should use a consistent responsive grid that can scale naturally to tablet and mobile breakpoints during implementation.

---

## Spacing

Maintain consistent spacing between:

- sections,
- components,
- text blocks.

Whitespace should communicate structure.

---

## Section Rhythm

Each section should feel visually separated while remaining connected to the overall narrative.

---

# Typography

Typography should be the primary communication tool.

Prioritize:

- strong hierarchy,
- readability,
- generous line height,
- restrained font weights.

Define:

- H1
- H2
- H3
- Body
- Small text

Typography should create rhythm before color does.

Spacing and typography should communicate hierarchy even in grayscale.

---

# Color Palette

At this stage define only:

- Primary
- Secondary
- Background
- Surface
- Border
- Text
- Muted Text

The palette should reinforce professionalism rather than attract attention.

Color should support hierarchy—not create it.

---

# Components

Define the visual style for:

- Navigation
- Primary Button
- Secondary Button
- Card
- Input
- Badge

Component styling should remain minimal and highly reusable.

Components should appear as members of the same system rather than individually designed elements.

---

# Imagery

Prefer:

- Product screenshots
- Workflow diagrams
- Interface previews
- Abstract supporting graphics

Avoid:

- Stock photography
- Decorative illustrations
- Cartoon artwork
- Visual elements unrelated to product communication

---

# Motion

Motion should be:

- subtle,
- intentional,
- fast,
- accessible.

Animation should support interaction rather than entertain.

---

# Design References

Reference products:

- Linear
- Vercel
- Stripe
- Raycast

Reference them for:

- typography,
- spacing,
- hierarchy,
- visual restraint,
- product-first communication.

They are references for design quality—not templates to copy.

Study principles, not appearances.

---

# Review Checklist

Before approving the visual direction, verify that:

- The design communicates professionalism.
- Visual hierarchy is immediately clear.
- Typography carries the interface.
- Whitespace improves readability.
- Components are visually consistent.
- The product remains the primary focus.
- Nothing distracts from the conversion flow.
- The direction can scale across future pages.
- The interface communicates quality before functionality is explored.

---

# Exit Condition

This artifact is complete when:

- The homepage can be designed without additional stylistic decisions.
- Future interfaces can reuse the same visual language.
- Approved product decisions remain unchanged.
- Visual consistency can be maintained across the product.
